

# Revised Plan: Help page — refinements before approval

Only the changed parts are restated. Everything else from the prior plan stands.

---

## 1. Content ownership model (single runtime source)

**Master product source**: `docs/product/capabilities.md` — unchanged. All public-eligible facts originate here.

**Runtime source for the Help page**: `src/content/help/faq.ts` (and siblings `workflow.ts`, `features.ts`, `troubleshooting.ts`). The app reads only from these.

**Treatment of `docs/product/faq.md`**: **deprecate as a manually maintained doc.** On the day Phase 3 ships, replace its body with a one-line pointer:

> Deprecated. The runtime FAQ now lives in `src/content/help/faq.ts`. The product source of truth is `docs/product/capabilities.md`. Do not edit this file.

We do **not** auto-generate `faq.md` from `faq.ts` in v1 — generation adds tooling without a current consumer. If a non-runtime editorial export is needed later, regenerate from `faq.ts`.

**Drift guard** (Phase 4): `scripts/check-help-faq-coverage.mjs` asserts every `Public copy eligible: yes` capability with `FAQ seeds:` has at least one matching entry in `faq.ts` (matched by capability id stored on each FAQ item).

**Rule (added to Phase 1 as a header comment in `faq.ts`)**:
"Runtime FAQ source. Capabilities live in docs/product/capabilities.md. Do not edit docs/product/faq.md."

## 2. Support / contact approach (conservative)

- **No response-time promises.** Remove "response expectations" wording entirely.
- **Email**: do not hardcode `support@whatsaid.app` in the plan. Before Phase 3 builds the contact card, read the codebase (`Footer.tsx`, `Privacy.tsx`, `Terms.tsx`, `RefundPolicy.tsx`, locale files) for any already-published support email and reuse exactly that. If none exists, render a generic "Contact us from your account settings" card with a link to `/settings` — no email at all — and surface a `// TODO: confirm support email` note.
- **Card content (final)**: one short line ("Can't find an answer? Reach out — we read every message.") + one CTA (mailto if confirmed, otherwise `/settings`). No SLAs, no hours, no channels list.

## 3. Page filter scope (tightened)

Renamed from "search-the-page" to **"Filter help"** in the UI (placeholder: "Filter help…").

**v1 scope (locked):**
- Filters: **FAQ items + Troubleshooting items + Feature cards** only.
- Does **not** filter: workflow steps, quick links chips, hero, privacy strip, contact card, app navigation.
- Implementation: client-side, debounced 150ms, case-insensitive substring match on `title + body + group` of those three datasets. No fuzzy lib, no highlighting in v1.
- Empty-state: when query yields zero matches, show "No matches. Browse sections below." and keep the rest of the page visible.

**Honest v1 alternative considered**: ship without the filter and rely on the TOC. Rejected — FAQ + troubleshooting will exceed ~40 items combined; a local filter is genuinely useful and cheap. We keep it.

## 4. Language-launch strategy (no mixed-language UI)

**Decision**: shell strings translated (EN/IT/FR) for nav entry points and section headings; **long-form content (workflow bodies, feature card bodies, FAQ Q&A, troubleshooting) ships in EN/IT/FR together — no English-only fallback inside translated shells.**

**Why**: a translated navbar leading into English paragraphs feels broken for IT/FR users. Mixing languages within the same page erodes the "premium" bar.

**How we get there without stalling**:
- Content data files use a `{ en, it, fr }` shape per long string from day one: `body: { en: "...", it: "...", fr: "..." }`.
- EN copy authored during Phase 2/3.
- IT and FR translations produced in the **same phase that introduces the section** — not deferred. Volume is bounded (~7 workflow + ~25 feature lines + ~50 FAQ + ~10 troubleshooting). Manageable in one pass per phase.
- Runtime helper `pickLocale(field, i18n.language)` returns the active locale, falling back to EN only if a key is genuinely missing (defensive, not a launch strategy).

**Result**: every supported UI language gets a fully coherent Help page on launch. No "EN-only for v1" caveat.

## 5. Minor wording / scope refinements

- **UI label**: "Filter help" (input) — not "search the page".
- **Feature guide group rename**: "Organise & translate" → **"Organise & languages"**. The card inside scopes translation strictly to "Generate the summary in another language" (CAP-014, conservative wording). Tag translation cache (CAP-013) stays internal.
- **Troubleshooting wording**: "Question fails or is empty" → **"If a question returns an unhelpful answer"** with body "Try rephrasing the question or shortening it. Edit the saved question and re-run, or ask a more specific follow-up." No success guarantee.
- **Navbar entry points (de-cluttered)**:
  - **Desktop**: add "Help" only inside the user dropdown (signed-in) **and** under the existing footer "Support" column (visible to everyone, signed-in or not). No new top-bar link.
  - **Mobile**: add "Help" only in the mobile menu sheet. Do not also add a separate icon in the mobile header.
  - **Signed-out users on desktop**: reach Help via footer only (no dropdown exists). This is sufficient — the homepage inline link + footer covers discoverability.
  - Result: each surface has exactly one Help entry point, no duplicates.
